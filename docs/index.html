Soroban-Encrypt Developer Guide

A decentralized off-chain secrets management framework secured by Stellar's Soroban smart contracts, Shamir's Secret Sharing, and ECIES.

Overview & Architecture

Soroban-Encrypt provides decentralized off-chain data encryption with programmable on-chain access controls. Unlike standard storage solutions, decryption keys are split into multiple shares and distributed across self-hosted storage nodes. Keys are reconstructed dynamically by authenticating against a Soroban smart contract.

1. Local SSS Encryption & Key Splitting
⬇ (ECIES encrypted dispatch)
2. Distributed Storage across N Nodes
⬇ (Client authorization challenge)
3. On-chain validation on Soroban RPC
⬇ (Shares reconstructed)
4. Successful Decryption

How Nodes Work

Storage nodes act as gatekeepers to key shares. Nodes run a Go HTTP REST API backed by a secure BoltDB key-value store.

1. Encryption & Storage Dispatch

When encrypting files, the symmetric AES key is divided using Shamir's Secret Sharing (SSS) into $N$ shares with threshold $T$. Each share is encrypted using ECIES (Elliptic Curve Integrated Encryption Scheme) using the static P-256 public key of each destination storage node, ensuring only that specific node can read the share.

2. Soroban Authorization Verification

To retrieve a share, a caller must submit the transaction XDR of a call to the smart contract's approve function. The node simulates the transaction envelope locally via Soroban JSON-RPC. If the contract executes successfully without panicking, authorization is granted, and the node returns the decrypted key share.

CLI Quickstart

Encrypt and distribute key shares using the CLI:

./client-bin encrypt \
  -file sensitive_data.json \
  -out sensitive_data.enc \
  -contract "C..." \
  -nodes "http://node1:8080,http://node2:8080,http://node3:8080" \
  -n 3 -t 2 \
  -meta metadata.json

Smart Contracts

To integrate with Soroban-Encrypt, developers write and deploy a Soroban smart contract implementing an approve function:

#[contractimpl]
impl AllowlistContract {
    pub fn approve(env: Env, object_id: Bytes, caller: Address) {
        // Authenticate the caller
        caller.require_auth();

        // Enforce custom business logic (e.g. check subscription status or nft ownership)
        let has_access = env.storage().persistent().has(&caller);
        if !has_access {
            panic!("Access denied: subscriber not authorized");
        }
    }
}

API Specification

Storage nodes expose simple REST endpoints for managing and retrieving key shares.

POST /store

Stores an encrypted key share on the node. Protected by an optional API key configured via STORE_API_KEY.

{
  "object_id": "document_123",
  "contract_id": "CAS3...ABCD",
  "ephemeral_pubkey": "04a6...",
  "ciphertext": "de78...",
  "nonce": "a7b9..."
}

POST /retrieve

Retrieves a stored key share. The node verifies the caller address's signature and simulates the Soroban transaction on-chain before returning the decrypted share.

{
  "object_id": "document_123",
  "contract_id": "CAS3...ABCD",
  "caller_address": "GBXX...",
  "signature": "8b5f...",
  "message": "challengesignedmsg...",
  "tx_xdr": "AAAAA..."
}

Go Integration Example

To retrieve a secret share programmatically, developers initialize a client instance and retrieve shares from the storage nodes:

package main

import (
	"context"
	"fmt"
	"github.com/teeyml/soroban-encrypt/client"
)

func main() {
	// Initialize client using metadata and Stellar private key seed
	sdk, err := client.NewClient("metadata.json", "S...")
	if err != nil {
		panic(err)
	}

	// Fetch shares and decrypt the file
	decryptedBytes, err := sdk.DecryptFile(context.Background(), "sensitive_data.enc")
	if err != nil {
		fmt.Printf("Decryption failed: %v\n", err)
		return
	}
	fmt.Printf("Decrypted secret content: %s\n", string(decryptedBytes))
}
